# Autenticação de API e Gerenciamento de Chaves do CertoPay

**Autentique toda solicitação à API CertoPay usando X-Api-Key ou tokens JWT Bearer. Abrange recuperação de chaves, rotação, limites de taxa e boas práticas de segurança.**

Toda solicitação à API CertoPay deve ser autenticada. O método principal é uma **Chave de API** passada no cabeçalho `X-Api-Key` — é stateless, fácil de rotacionar e a escolha certa para todas as integrações servidor a servidor. Um **token JWT Bearer** também é suportado para operações de gerenciamento, como rotação de chaves, quando você prefere credenciais de curta duração.

---

## Chave de API (Recomendado)

As Chaves de API são a forma padrão de autenticar chamadas à API CertoPay a partir do seu backend. Inclua a chave no cabeçalho `X-Api-Key` em toda solicitação.

```bash
curl https://v2.certopaybrasil.com/api/customers \
  -H "X-Api-Key: sk_live_sua_chave_aqui"
```

> ⚠️ **Atenção:** Nunca exponha sua Chave de API em JavaScript de frontend, binários de aplicativos móveis, extensões de navegador ou repositórios públicos. A chave concede acesso total à sua conta CertoPay — trate-a com o mesmo cuidado de uma senha de banco de dados. Sempre passe-a a partir do seu servidor e armazene-a em uma variável de ambiente (ex.: `CERTOPAY_API_KEY`), nunca no código-fonte.

### Recuperar Sua Chave Atual

Você pode buscar metadados sobre sua Chave de API ativa de forma programática. Por ser uma operação de gerenciamento, ela requer um token JWT Bearer de curta duração (obtido pelo endpoint de login abaixo) em vez da própria Chave de API.

```http
GET /api/api-keys/me
Authorization: Bearer {jwt_token}
```

**Resposta — `200 OK`**

```json
{
  "id": "uuid",
  "label": "Chave principal",
  "keyPrefix": "sk_live_a1b2c3d4...",
  "maskedKey": "sk_live_a1b2c3d4...••••••••••",
  "active": true,
  "rateLimit": 100,
  "lastUsedAt": "2026-06-26T10:00:00Z",
  "createdAt": "2026-06-01T10:00:00Z"
}
```

| Campo | Tipo | Descrição |
|---|---|---|
| `id` | string | UUID do registro da Chave de API. |
| `label` | string | Rótulo legível atribuído à chave no painel. |
| `keyPrefix` | string | O prefixo visível da chave (primeiros 16 caracteres). Use para identificar qual chave está ativa sem expor o segredo. |
| `maskedKey` | string | Versão parcialmente mascarada da chave, apenas para exibição. |
| `active` | boolean | Indica se a chave está ativa. Chaves inativas retornam `401 Unauthorized`. |
| `rateLimit` | integer | Número máximo de solicitações por minuto permitidas para esta chave. |
| `lastUsedAt` | string | Timestamp ISO 8601 da solicitação autenticada mais recente feita com esta chave. |
| `createdAt` | string | Timestamp ISO 8601 de quando a chave foi criada originalmente. |

> ⚠️ **Atenção:** A **chave secreta completa** é exibida apenas uma vez — imediatamente após a criação no painel. Se você perder sua chave antes de salvá-la, será necessário regenerá-la. Não há como recuperar o valor original.

### Regenerar Sua Chave

A regeneração emite um novo segredo e invalida imediatamente o anterior. Quaisquer solicitações em andamento usando a chave anterior falharão com `401`. Atualize todas as suas integrações antes de regenerar em produção.

```http
POST /api/api-keys/me/regenerate
Authorization: Bearer {jwt_token}
```

**Resposta — `200 OK`**

```json
{
  "id": "uuid",
  "label": "Chave principal",
  "fullKey": "sk_live_new_full_secret_here",
  "active": true,
  "createdAt": "2026-06-26T12:00:00Z"
}
```

> 💡 **Dica:** Antes de rotacionar uma chave em produção, implante a nova chave em todos os seus serviços primeiro e só então chame o endpoint de regeneração. Isso minimiza o tempo de inatividade para os poucos segundos entre a chamada de rotação e os serviços carregando o novo valor.

## Token JWT Bearer (Alternativo)

Os tokens JWT são credenciais de curta duração adequadas para operações de gerenciamento — recuperar metadados de chaves, regenerar chaves e acessar a API do painel do vendedor. Não são recomendados como substituto da Chave de API em fluxos de pagamento.

### Obter um Token

```http
POST /api/auth/login
Content-Type: application/json
```

```json
{
  "email": "seu@email.com",
  "password": "sua_senha"
}
```

**Resposta — `200 OK`**

```json
{
  "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}
```

`accessToken` — Um JWT assinado. Inclua-o no cabeçalho `Authorization` das solicitações de gerenciamento. Os tokens expiram após um curto período — verifique a claim `exp` no payload.

### Usar o Token

```bash
curl https://v2.certopaybrasil.com/api/api-keys/me \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
```

> ℹ️ **Nota:** JWTs não são aceitos nos endpoints de processamento de pagamento (`/api/payment-gateway/process`, `/api/orders`, `/api/customers`). Use seu `X-Api-Key` para essas chamadas.

## Limites de Taxa

> ℹ️ **Nota:** O limite de taxa padrão é de **100 solicitações por minuto** por Chave de API. Quando o limite é excedido, a API retorna `HTTP 429 Too Many Requests`. As solicitações são contabilizadas em uma janela deslizante de 60 segundos. Se você precisar de um limite maior para o seu caso de uso, entre em contato com o [suporte CertoPay](mailto:suporte@certopaybrasil.com) para solicitar a revisão e o aumento do seu limite.

Uma resposta `429` inclui um cabeçalho `Retry-After` indicando o número de segundos a aguardar antes de tentar novamente:

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 12
Content-Type: application/json

{
  "error": "rate_limit_exceeded",
  "message": "Too many requests. Please wait before retrying.",
  "retryAfter": 12
}
```

## Boas Práticas de Segurança

Seguir estas práticas protegerá sua Chave de API e os dados dos seus clientes.

**Use Variáveis de Ambiente** — Armazene sua Chave de API em uma variável de ambiente como `CERTOPAY_API_KEY` e leia-a em tempo de execução. Nunca a codifique diretamente nos arquivos de código-fonte.

**Separe Chaves de Teste e Produção** — Use uma chave dedicada para seu ambiente de staging/teste e uma chave separada para produção. Isso evita cobranças acidentais durante o desenvolvimento.

**Restrinja a Exposição da Chave** — A chave deve residir apenas no seu servidor. Nunca a inclua em um aplicativo móvel, script de navegador ou qualquer artefato que possa ser descompilado ou inspecionado por usuários finais.

**Rotacione Chaves Periodicamente** — Rotacione sua Chave de API de produção periodicamente e imediatamente caso suspeite que ela foi comprometida. Use o endpoint de regeneração e implante a nova chave sem tempo de inatividade.

### Exemplo de Variável de Ambiente

```bash Node.js
# .env (nunca faça commit deste arquivo)
CERTOPAY_API_KEY=sk_live_sua_chave_aqui

# Na sua aplicação
const apiKey = process.env.CERTOPAY_API_KEY;
```

```python Python
# .env (nunca faça commit deste arquivo)
CERTOPAY_API_KEY=sk_live_sua_chave_aqui

# Na sua aplicação
import os
api_key = os.environ["CERTOPAY_API_KEY"]
```

```bash Docker / CI
# Passe como variável de ambiente em tempo de execução
docker run -e CERTOPAY_API_KEY=sk_live_sua_chave_aqui myapp

# Ou use um gerenciador de segredos (AWS Secrets Manager, HashiCorp Vault, etc.)
```

## Referência de Erros

| Status HTTP | Código | Significado |
|---|---|---|
| `401 Unauthorized` | `invalid_api_key` | Chave ausente, malformada ou revogada |
| `401 Unauthorized` | `expired_token` | Token JWT Bearer expirado — reautentique-se |
| `403 Forbidden` | `insufficient_permissions` | Chave existe, mas não tem permissão para esta operação |
| `429 Too Many Requests` | `rate_limit_exceeded` | Taxa de solicitações excedida — aguarde e tente novamente |
