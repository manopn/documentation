Sempre que uma transação avança em seu ciclo de vida — desde sua criação até o pagamento, reembolso ou expiração — a CertoPay envia um evento de webhook correspondente para o endpoint cadastrado. Compreender cada evento e saber qual ação tomar em resposta é essencial para criar uma integração confiável e resistente a fraudes. Esta página documenta todos os tipos de eventos que você pode receber, incluindo exemplos de payload e a lógica de tratamento recomendada para cada um.

---

## Todos os Eventos Disponíveis

A tabela abaixo lista todos os tipos de eventos que a CertoPay pode enviar. Cada nome de evento segue a convenção `resource.action` e representa uma transição específica no ciclo de vida da transação.

| Evento                   | Descrição                                                             |
| ------------------------ | --------------------------------------------------------------------- |
| `transaction.created`    | Uma nova transação foi criada e está aguardando pagamento.            |
| `transaction.paid`       | O pagamento foi confirmado e a transação foi liquidada com sucesso.   |
| `transaction.refused`    | A tentativa de pagamento foi recusada pelo processador ou emissor.    |
| `transaction.refunded`   | Um reembolso total ou parcial foi processado com sucesso.             |
| `transaction.chargeback` | O comprador contestou a cobrança e um chargeback foi iniciado.        |
| `transaction.expired`    | Um QR Code PIX ou Boleto expirou sem que o pagamento fosse realizado. |
| `transaction.cancelled`  | A transação foi cancelada antes da captura do pagamento.              |

<Note>
  Todos os eventos compartilham a mesma estrutura de envelope JSON. Apenas o valor do campo `event` e alguns campos dentro do objeto `data` variam entre os diferentes tipos de evento. Consulte a página **Visão Geral de Webhooks — Formato do Payload** para uma descrição completa da estrutura do webhook.
</Note>

## Exemplos de Payload

Os exemplos abaixo mostram payloads representativos dos eventos mais comuns. Utilize-os para desenvolver e testar o tratamento dos webhooks antes de colocar sua integração em produção.

### `transaction.paid`

Enviado quando um pagamento é totalmente confirmado. Este é o evento que deve ser utilizado antes de liberar acesso a um produto ou serviço.

```json
{
  "event": "transaction.paid",
  "created_at": "2026-06-26T10:05:00Z",
  "data": {
    "transactionId": "uuid-da-transacao",
    "orderId": "uuid-do-pedido",
    "amount": 29700,
    "method": "PIX",
    "status": "PAID",
    "paidAt": "2026-06-26T10:05:00Z"
  }
}
```

---

### `transaction.refused`

Enviado quando uma tentativa de pagamento é recusada. As causas mais comuns incluem saldo insuficiente, dados incorretos do cartão ou regras antifraude acionadas pelo emissor.

```json
{
  "event": "transaction.refused",
  "created_at": "2026-06-26T10:05:00Z",
  "data": {
    "transactionId": "uuid-da-transacao",
    "orderId": "uuid-do-pedido",
    "amount": 29700,
    "method": "PIX",
    "status": "REFUSED"
  }
}
```

---

### `transaction.refunded`

Enviado quando um reembolso é processado e os valores estão sendo devolvidos ao comprador.

```json
{
  "event": "transaction.refunded",
  "created_at": "2026-06-26T10:05:00Z",
  "data": {
    "transactionId": "uuid-da-transacao",
    "orderId": "uuid-do-pedido",
    "amount": 29700,
    "method": "PIX",
    "status": "REFUNDED"
  }
}
```

---

### `transaction.expired`

Enviado quando um QR Code PIX ou código de barras do boleto ultrapassa sua data de expiração sem a confirmação do pagamento.

```json
{
  "event": "transaction.expired",
  "created_at": "2026-06-26T10:05:00Z",
  "data": {
    "transactionId": "uuid-da-transacao",
    "orderId": "uuid-do-pedido",
    "amount": 29700,
    "method": "PIX",
    "status": "EXPIRED"
  }
}
```

## Como Tratar Cada Evento

Cada evento exige uma resposta diferente da sua aplicação. As orientações abaixo descrevem a ação recomendada para cada tipo de evento. Implemente tratadores para todos os eventos relevantes ao seu negócio, mesmo que a ação inicial seja apenas registrar a alteração de status.

<Accordion title="transaction.created — Nova transação aguardando pagamento">
  **Ação recomendada:** Registre a transação em seu banco de dados e exiba ao comprador as instruções de pagamento correspondentes (por exemplo, o QR Code PIX ou o código de barras do boleto). Nenhuma entrega ou liberação deve ocorrer neste momento.

Este evento confirma que a transação foi criada no sistema da CertoPay e encontra-se em estado pendente. Para métodos de pagamento síncronos (como cartão de crédito), este evento pode ser seguido quase imediatamente por `transaction.paid` ou `transaction.refused`. </Accordion>

<Accordion title="transaction.paid — Pagamento confirmado ✅">
  **Ação recomendada:** Confirme a transação no servidor realizando uma chamada para `GET /api/transactions/{transactionId}` e, somente após essa validação, conceda acesso ao produto ou serviço adquirido.

Este é o evento mais importante de todo o ciclo de vida da transação. As ações de entrega normalmente incluem:

* Ativar uma assinatura ou licença
* Liberar um download digital
* Marcar o pedido como "pago" e iniciar o envio de produtos físicos
* Enviar um e-mail de confirmação da compra

  <Warning>
    Sempre confirme o status da transação através da API antes de realizar qualquer entrega. Nunca conceda acesso apenas com base no payload recebido via webhook. Consulte a seção **Verificando Eventos** para conhecer o fluxo completo de validação.
  </Warning>
</Accordion>

<Accordion title="transaction.refused — Pagamento recusado ❌">
  **Ação recomendada:** Informe ao comprador que o pagamento não foi aprovado e ofereça uma forma simples de tentar novamente.

Boas práticas:

* Exiba uma mensagem amigável ao usuário (evite expor códigos internos do processador)
* Sugira métodos de pagamento alternativos — por exemplo, oferecer PIX quando um cartão for recusado
* **Não** marque imediatamente o pedido como definitivamente falho; o comprador pode tentar novamente utilizando outra forma de pagamento em uma nova transação
* Registre o motivo da recusa para análises internas e monitoramento antifraude

  <Tip>
    Para recusas em cartão de crédito, o objeto `data` pode incluir o campo `refusalReason`, contendo um código retornado pelo processador. Consulte a Referência da API para obter a lista completa dos códigos de recusa.
  </Tip>
</Accordion>

<Accordion title="transaction.refunded — Reembolso processado 🔄">
  **Ação recomendada:** Revogue ou suspenda o acesso do comprador ao produto ou serviço e envie uma confirmação de reembolso.

Etapas recomendadas:

* Atualize o pedido para o status `REFUNDED` em seu banco de dados
* Revogue o acesso: desative licenças, bloqueie contas ou cancele assinaturas
* Envie um e-mail informando o prazo estimado para devolução dos valores (reembolsos via PIX normalmente são imediatos; reembolsos em cartão podem levar alguns dias úteis)
* Atualize registros de estoque ou de entrega, quando aplicável

  <Note>
    O reembolso pode ser total ou parcial. Utilize `data.amount` para identificar o valor reembolsado e compare-o com o valor original da transação para determinar qual dos dois casos ocorreu.
  </Note>
</Accordion>

<Accordion title="transaction.chargeback — Chargeback iniciado ⚠️">
  **Ação recomendada:** Sinalize imediatamente a transação para sua equipe de prevenção a fraudes, suspenda o acesso do comprador e reúna todas as evidências necessárias para responder à contestação pelo painel da CertoPay.

O chargeback representa uma contestação formal iniciada pelo titular do cartão junto ao banco emissor. O tempo é um fator crítico, pois as bandeiras estabelecem prazos rigorosos para apresentação da defesa.

Recomenda-se:

* Suspender ou revogar imediatamente o acesso ao produto ou serviço relacionado ao pedido
* Marcar a conta do comprador para revisão antifraude
* Registrar todas as informações da transação, entrega e comunicações que possam servir como evidência
* Monitorar o painel da CertoPay para acompanhar os prazos e responder rapidamente

  <Warning>
    Chargebacks que não recebem resposta são automaticamente decididos em favor do comprador. Responda a todas as contestações dentro do prazo estabelecido, mesmo quando acreditar que sejam fraudulentas.
  </Warning>
</Accordion>

<Accordion title="transaction.expired — PIX ou boleto expirado 🕐">
  **Ação recomendada:** Marque o pedido correspondente como expirado e oriente o comprador a reiniciar o processo de pagamento.

QR Codes PIX e boletos possuem um período de validade limitado. Após a expiração, o pagamento não poderá mais ser realizado utilizando as instruções originais.

Etapas recomendadas:

* Atualize o pedido para o status `EXPIRED`
* Envie uma notificação ao comprador com um link para reiniciar o checkout
* Não gere automaticamente uma nova transação; permita que o comprador decida quando tentar novamente
* Opcionalmente, libere qualquer estoque que tenha sido reservado temporariamente para esse pedido

  </Accordion>

<Accordion title="transaction.cancelled — Transação cancelada 🚫">
  **Ação recomendada:** Marque o pedido como cancelado em seu sistema. Nenhum pagamento foi recebido e, portanto, nenhum reembolso é necessário.

Diferentemente das recusas e expirações, um cancelamento representa o encerramento explícito da transação antes da captura do pagamento. Isso pode ocorrer por uma ação da sua aplicação (como uma chamada à API para cancelamento) ou por regras internas da CertoPay.

Etapas recomendadas:

* Atualize o pedido para o status `CANCELLED`
* Libere qualquer estoque reservado temporariamente
* Notifique o comprador caso o cancelamento não tenha sido iniciado por ele
* Não tente cobrar novamente utilizando a mesma transação; crie uma nova transação, se necessário

  </Accordion>

## Garantias de Entrega dos Eventos

<Note>
  A CertoPay garante a entrega dos eventos utilizando o modelo **at-least-once delivery** (entrega de pelo menos uma vez). Por esse motivo, seu endpoint de webhook deve ser **idempotente**: processar o mesmo evento mais de uma vez não deve gerar entregas duplicadas, reembolsos duplicados ou qualquer outro efeito colateral indesejado. Utilize `data.transactionId` como chave de idempotência ao registrar alterações de estado em seu banco de dados.
</Note>

Para mais informações sobre novas tentativas de entrega, histórico de eventos e reenvio manual, consulte a página **Visão Geral de Webhooks — Tratando Entregas**.
