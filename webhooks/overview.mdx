Os webhooks são a principal forma de a CertoPay comunicar alterações no ciclo de vida dos pagamentos à sua aplicação. Em vez de consultar repetidamente a API para verificar se uma transferência PIX foi compensada ou um boleto foi pago, você registra um endpoint HTTPS em seu servidor, e a CertoPay envia uma requisição HTTP `POST` assinada assim que o status da transação muda. Isso é especialmente importante para métodos de pagamento assíncronos, como **PIX** e **Boleto**, cuja confirmação pode ocorrer segundos ou até horas após o checkout.

## O que são Webhooks?

Quando o status de uma transação muda — por exemplo, quando um pagamento via PIX é confirmado — a CertoPay envia uma requisição HTTP `POST` contendo um payload JSON para a URL cadastrada. Seu servidor processa esse payload e responde com um HTTP `200` para confirmar o recebimento.

<Note>
  Pagamentos via **PIX** e **Boleto** são, por natureza, **assíncronos**. O checkout cria a transação e apresenta as instruções de pagamento ao comprador, mas a confirmação do pagamento chega posteriormente por meio do webhook. Sem um endpoint de webhook configurado, seu sistema não terá uma forma confiável de saber quando esses pagamentos foram compensados.
</Note>

Características dos webhooks da CertoPay:

* São enviados como requisições `HTTP POST` com `Content-Type: application/json`
* Incluem um tipo de evento (`event`), um timestamp e um objeto `data` com os detalhes da transação
* São reenviados automaticamente caso seu endpoint não responda com `HTTP 200`
* Todo o histórico de entregas pode ser consultado através do endpoint de histórico

## Registrar ou Atualizar um Webhook

Sua conta pode possuir apenas uma URL de webhook ativa por vez. A chamada ao endpoint de *upsert* registra uma nova URL ou substitui a existente.

```http
POST /api/webhooks/upsert
X-Api-Key: sk_live_sua_chave_aqui
Content-Type: application/json

{
  "url": "https://seusite.com/webhooks/certopay"
}
```

**Corpo da requisição**

| Campo | Tipo   | Obrigatório | Descrição                                                        |
| ----- | ------ | ----------- | ---------------------------------------------------------------- |
| `url` | string | ✅ Sim       | URL pública HTTPS que receberá as requisições POST dos webhooks. |

<Tip>
  Utilize sempre **HTTPS**. A CertoPay não envia eventos para endpoints HTTP. Certifique-se de que seu certificado SSL é válido e que seu servidor responde com `HTTP 200` sem redirecionamentos — redirecionamentos não são seguidos.
</Tip>

## Consultar o Webhook Registrado

Recupere a URL de webhook atualmente cadastrada para sua conta.

```http
GET /api/webhooks
X-Api-Key: sk_live_sua_chave_aqui
```

**Exemplo de resposta**

```json
{
  "url": "https://seusite.com/webhooks/certopay",
  "createdAt": "2026-06-01T12:00:00Z",
  "updatedAt": "2026-06-20T08:30:00Z"
}
```

## Desabilitar o Webhook

Enviar uma requisição `DELETE` remove o endpoint de webhook cadastrado. Nenhum outro evento será enviado até que uma nova URL seja registrada.

```http
DELETE /api/webhooks
X-Api-Key: sk_live_sua_chave_aqui
```

Uma exclusão bem-sucedida retorna `HTTP 204 No Content`.

<Warning>
  Se você remover seu webhook enquanto existirem transações PIX ou boletos ainda pendentes de confirmação, **não receberá as confirmações de pagamento** dessas transações. Antes de desabilitar o endpoint, certifique-se de possuir outro processo de conciliação.
</Warning>

## Formato do Payload do Webhook

Todos os eventos enviados pela CertoPay compartilham a mesma estrutura principal (*envelope*), independentemente do tipo de evento. O objeto `data` contém as informações específicas da transação que originou o evento.

```json
{
  "event": "transaction.paid",
  "created_at": "2026-06-26T10:05:00Z",
  "data": {
    "transactionId": "uuid-da-transacao",
    "orderId": "uuid-do-pedido",
    "amount": 29700,
    "method": "PIX",
    "status": "PAID",
    "paidAt": "2026-06-26T10:05:00Z"
  }
}
```

**Campos do nível superior**

| Campo        | Tipo   | Descrição                                                        |
| ------------ | ------ | ---------------------------------------------------------------- |
| `event`      | string | Identificador do tipo de evento, por exemplo `transaction.paid`. |
| `created_at` | string | Data e hora em UTC (ISO 8601) em que o evento foi gerado.        |
| `data`       | object | Informações da transação relacionadas ao evento.                 |

**Campos do objeto `data`**

| Campo           | Tipo    | Descrição                                                                                                       |
| --------------- | ------- | --------------------------------------------------------------------------------------------------------------- |
| `transactionId` | string  | Identificador único da transação (UUID).                                                                        |
| `orderId`       | string  | Identificador do pedido associado (UUID).                                                                       |
| `amount`        | integer | Valor da transação em **centavos** (ex.: `29700` = R$ 297,00).                                                  |
| `method`        | string  | Método de pagamento: `PIX`, `BOLETO` ou `CREDIT_CARD`.                                                          |
| `status`        | string  | Status atual da transação (consulte a página **Eventos**).                                                      |
| `paidAt`        | string  | Data e hora em UTC (ISO 8601) em que o pagamento foi confirmado. Presente apenas em eventos `transaction.paid`. |

## Tratando as Entregas

### Respondendo à CertoPay

Seu endpoint de webhook deve responder com um **HTTP `200`** em até **5 segundos** após receber a requisição. Qualquer outro código de resposta, ou uma resposta superior a 5 segundos, será considerada uma falha na entrega e o evento será agendado para uma nova tentativa.

<Note>
  O limite de **5 segundos** é rigoroso. Caso seu processamento envolva operações demoradas — como gravação em banco de dados, envio de e-mails ou chamadas para APIs externas — responda imediatamente com `HTTP 200` e processe o evento de forma assíncrona em uma fila ou tarefa em segundo plano.
</Note>

### Novas Tentativas Automáticas

Quando uma entrega falha, a CertoPay agenda automaticamente novas tentativas utilizando uma estratégia de **backoff exponencial**. As tentativas continuam até que seu endpoint responda com `HTTP 200` ou o período máximo de tentativas seja atingido.

Por esse motivo, seu endpoint deve ser **idempotente**, pois o mesmo evento poderá ser entregue mais de uma vez.

### Consultando o Histórico de Entregas

Você pode visualizar todas as tentativas realizadas para seu endpoint, incluindo detalhes da requisição, resposta e status.

```http
GET /api/webhooks/deliveries
X-Api-Key: sk_live_sua_chave_aqui
```

**Filtrar por status da entrega**

| Filtro            | Descrição                                                                      |
| ----------------- | ------------------------------------------------------------------------------ |
| `?status=SUCCESS` | Entregas confirmadas com `HTTP 200`.                                           |
| `?status=FAILED`  | Entregas que retornaram um código diferente de `200` ou expiraram por timeout. |
| `?status=PENDING` | Entregas aguardando a primeira tentativa ou um novo reenvio.                   |

**Filtrar por tipo de evento**

```http
GET /api/webhooks/deliveries?event=transaction.paid
X-Api-Key: sk_live_sua_chave_aqui
```

### Reenviando uma Entrega Manualmente

Caso uma entrega tenha falhado e você não queira aguardar a próxima tentativa automática, utilize o endpoint de reenvio manual.

```http
POST /api/webhooks/deliveries/{deliveryId}/retry
X-Api-Key: sk_live_sua_chave_aqui
```

Substitua `{deliveryId}` pelo identificador retornado pelo endpoint de histórico de entregas.

<Accordion title="Exemplo de resposta do histórico de entregas">

```json
{
  "deliveries": [
    {
      "deliveryId": "dlv_abc123",
      "event": "transaction.paid",
      "status": "FAILED",
      "httpStatus": 500,
      "attemptedAt": "2026-06-26T10:05:03Z",
      "nextRetryAt": "2026-06-26T10:10:03Z"
    },
    {
      "deliveryId": "dlv_def456",
      "event": "transaction.paid",
      "status": "SUCCESS",
      "httpStatus": 200,
      "attemptedAt": "2026-06-26T10:10:05Z",
      "nextRetryAt": null
    }
  ]
}
```

</Accordion>

## Verificando Eventos

Receber um webhook, por si só, não é suficiente para confirmar um pagamento. O payload pode ser reenviado, falsificado (*spoofing*) ou representar um estado intermediário da transação. Antes de conceder acesso a qualquer produto ou serviço, sempre confirme o status diretamente através da API da CertoPay.

```http
GET /api/transactions/{transactionId}
X-Api-Key: sk_live_sua_chave_aqui
```

Somente prossiga com a entrega do produto ou serviço (como liberar um produto digital ou ativar uma assinatura) quando a API confirmar que a transação possui `status: "PAID"`.

<Warning>
  **Nunca conceda acesso baseando-se apenas no payload recebido pelo webhook.** Sempre faça uma chamada no servidor para `GET /api/transactions/{transactionId}` e confirme o status atual antes de concluir o pedido. Isso protege sua aplicação contra ataques de replay e condições de corrida (*race conditions*), nas quais uma alteração posterior (como um chargeback) pode invalidar um evento anterior de `transaction.paid`.
</Warning>

**Fluxo recomendado de verificação**

1. Receba a requisição `POST` no endpoint do webhook.
2. Extraia o `transactionId` de `data.transactionId`.
3. Faça uma chamada para `GET /api/transactions/{transactionId}` utilizando sua chave de API.
4. Verifique se o `status` retornado corresponde ao esperado (por exemplo, `PAID`).
5. Conclua o pedido em seu sistema (liberação do produto ou serviço).
6. Responda com `HTTP 200` para a CertoPay.
