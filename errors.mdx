Quando algo dá errado, a CertoPay retorna uma resposta de erro JSON estruturada junto com o código de status HTTP apropriado. Esta página é sua referência para entender o que cada código de status e mensagem de erro significa, e qual ação tomar para resolvê-lo.

## Formato da Resposta de Erro

Toda resposta de erro da API CertoPay segue a mesma estrutura JSON:

```json
{
  "statusCode": 400,
  "message": "Descrição do erro",
  "error": "Bad Request"
}
```

| Campo | Tipo | Descrição |
|---|---|---|
| `statusCode` | `number` | O código de status HTTP do erro |
| `message` | `string` | Uma descrição legível do que deu errado |
| `error` | `string` | A frase HTTP padrão correspondente ao código |

Use o campo `message` para depuração e logs. Não o exiba diretamente aos usuários finais sem sanitização.

## Referência de Códigos de Status HTTP

A tabela abaixo lista todos os códigos de status HTTP que você pode encontrar e a ação recomendada para cada um.

| Status | Significado | Como Resolver |
|---|---|---|
| `400` | Payload inválido ou parâmetro ausente | Verifique os campos obrigatórios no corpo da requisição em relação à documentação do endpoint. Consulte o campo `message` para identificar o campo específico que falhou na validação. |
| `401` | Chave de API ou JWT ausente ou inválido | Verifique se o header `X-Api-Key` está presente na requisição e se o valor da chave está correto. Certifique-se de estar usando a chave correta para o ambiente (teste vs. produção). |
| `403` | Permissões insuficientes | Confirme se a Chave de API utilizada possui as permissões necessárias para o recurso ou ação que você está tentando executar. |
| `404` | Recurso não encontrado | Verifique o valor do ID no caminho da URL. Certifique-se de que o recurso existe e pertence à mesma conta da Chave de API. |
| `429` | Limite de requisições excedido | Sua integração está enviando mais de 100 requisições por minuto. Reduza o ritmo das requisições e implemente backoff exponencial. Consulte [Segurança e Limites de Requisição](/best-practices/security#rate-limits). |
| `500` | Erro interno do servidor | Ocorreu um erro inesperado no lado da CertoPay. Tente novamente após um breve intervalo. Se o erro persistir, entre em contato com o suporte informando os detalhes da requisição. |

## Erros Comuns de Integração

Os erros abaixo representam a maioria dos problemas de integração relatados durante o desenvolvimento e a entrada em produção. Expanda cada um para ver a descrição e os passos de resolução.

<AccordionGroup>

  <Accordion title='"buyerDoc is required"'>
    **O que significa:** O método de pagamento selecionado (PIX ou Boleto) exige o CPF ou CNPJ do comprador para gerar a cobrança. O campo `buyerDoc` estava ausente no corpo da requisição.

    **Como resolver:** Adicione o campo `buyerDoc` ao corpo da sua requisição `POST /api/payment-gateway/process`, preenchido com o CPF (11 dígitos) ou CNPJ (14 dígitos) do comprador:

    ```json
    {
      "orderId": "uuid-pedido",
      "method": "PIX",
      "amount": 29700,
      "buyerDoc": "12345678901"
    }
    ```

    <Note>
      `buyerDoc` é obrigatório para PIX e Boleto. Não é necessário para pagamentos com cartão de crédito, onde os próprios dados do cartão identificam o pagador.
    </Note>
  </Accordion>

  <Accordion title='"orderId not found"'>
    **O que significa:** O valor de `orderId` na sua requisição não corresponde a nenhum registro de pedido na CertoPay, ou o pedido existe mas pertence a uma conta diferente da Chave de API utilizada.

    **Como resolver:**
    - Verifique se o `orderId` foi salvo corretamente e está sendo lido do registro de banco de dados correto.
    - Confirme se você está usando a Chave de API correta para o ambiente onde o pedido foi criado (teste vs. produção).
    - Se você está consultando um pedido existente, certifique-se de que ele não foi excluído ou expirado.
  </Accordion>

  <Accordion title='"Idempotency key already used with different payload"'>
    **O que significa:** Você enviou uma requisição com um `Idempotency-Key` que já foi usado em uma requisição anterior nas últimas 24 horas, mas o corpo da requisição não corresponde ao original. A CertoPay rejeita isso para evitar tentativas de cobrança ambíguas ou conflitantes.

    **Como resolver:** Gere um novo UUID v4 e use-o como `Idempotency-Key` para esta requisição. Se você pretendia repetir a cobrança original, restaure o corpo da requisição exatamente como foi enviado na primeira vez.

    ```javascript
    import { v4 as uuidv4 } from 'uuid';
    const newKey = uuidv4(); // gera uma chave totalmente nova
    ```

    Consulte [Idempotência](/best-practices/idempotency) para a estratégia completa de tentativas.
  </Accordion>

  <Accordion title="Transação permanece no status PENDING">
    **O que significa:** Para pagamentos via PIX e Boleto, o status `PENDING` é **esperado e normal**. Esses métodos de pagamento exigem que o comprador tome uma ação separada — escanear o QR Code ou pagar o boleto — antes que a transação seja confirmada. O status permanecerá `PENDING` até que o comprador conclua o pagamento.

    **Como resolver:** Não faça polling do status da transação em loop contínuo. Em vez disso:
    1. Apresente o QR Code PIX ou o código de barras do Boleto ao comprador.
    2. Aguarde o evento de webhook `transaction.paid` no seu endpoint.
    3. Quando o webhook chegar, verifique o status via `GET /api/transactions/{transactionId}` e confirme o pedido quando o `status` for `PAID`.

    <Note>
      Pagamentos PIX geralmente são confirmados em segundos. Pagamentos via Boleto podem levar até **3 dias úteis** para serem confirmados após o pagamento em um banco ou terminal de pagamento.
    </Note>
  </Accordion>

  <Accordion title="HTTP 429 em POST /api/payment-gateway/process">
    **O que significa:** Sua integração excedeu o limite padrão de 100 requisições por minuto para sua Chave de API. Todas as requisições seguintes estão sendo rejeitadas até que a janela de limite seja redefinida.

    **Como resolver:**
    - Adicione um intervalo entre as requisições de pagamento se estiver processando-as em loop ou em lote.
    - Implemente backoff exponencial para que seu código aguarde progressivamente mais tempo após cada resposta `429` antes de tentar novamente.
    - Revise sua integração em busca de chamadas duplicadas não intencionais — por exemplo, um loop de tentativas que não respeita o ciclo de vida da chave de idempotência.
    - Se sua necessidade legítima de throughput exceder 100 req/min, entre em contato com o suporte da CertoPay para solicitar um limite maior para sua conta.

    Consulte [Limites de Requisição](/best-practices/security#rate-limits) para um exemplo de código com backoff exponencial.
  </Accordion>

</AccordionGroup>

<Tip>

Essas informações ajudam a equipe de suporte a diagnosticar o problema rapidamente.
</Tip>
