# Segurança da API CertoPay: Armazenamento de Chaves e Boas Práticas

**Subtítulo da barra lateral:** Segurança

**Descrição:** Mantenha sua Chave de API do CertoPay segura. Cobre armazenamento de chaves, rotação, verificação de webhooks e tratamento de limites de taxa para integrações em produção.

Sua Chave de API do CertoPay é a credencial mestra que autoriza toda operação de pagamento e de dados em sua conta. Qualquer pessoa que a possua pode iniciar cobranças, consultar transações e modificar sua integração. Esta página cobre os passos essenciais para manter essa chave — e sua integração como um todo — segura em produção.

## Protegendo Sua Chave de API

A causa mais comum de comprometimento de uma chave de API é a exposição acidental: enviar uma chave para o controle de versão, incorporá-la em JavaScript do lado do cliente ou registrá-la em texto puro nos logs. Siga estas regras para evitar isso.

### Nunca exponha sua chave no lado do cliente

Sua Chave de API deve existir apenas no seu **servidor**. Ela nunca deve aparecer em:
- JavaScript de navegador (React, Vue, vanilla JS, etc.)
- Aplicativos móveis nativos ou híbridos (iOS, Android, React Native, Flutter)
- Repositórios Git públicos ou privados (mesmo no histórico de commits)
- Artefatos de build, imagens Docker ou logs de CI/CD

> **Aviso:** Se sua Chave de API for acidentalmente enviada a um repositório ou exposta em algum local público, **rotacione-a imediatamente** usando `POST /api/api-keys/me/regenerate`. Trate qualquer janela de exposição como um comprometimento confirmado.

### Armazene chaves em variáveis de ambiente

Nunca codifique sua Chave de API como um literal de string diretamente no código-fonte. Carregue-a a partir de uma variável de ambiente em tempo de execução:

**.env**
```bash
CERTOPAY_API_KEY=sk_live_sua_chave_aqui
```

**Node.js**
```javascript
const apiKey = process.env.CERTOPAY_API_KEY;
```

**Python**
```python
import os

api_key = os.environ["CERTOPAY_API_KEY"]
```

Para ambientes hospedados, use o sistema de gerenciamento de segredos da sua plataforma — AWS Secrets Manager, GCP Secret Manager, Vercel Environment Variables, Railway secrets, etc. — em vez de um arquivo `.env` simples em disco.

> **Nota:** Adicione `.env` ao seu arquivo `.gitignore` para garantir que ele nunca seja enviado ao controle de versão.

### Use chaves separadas para teste e produção

O CertoPay fornece chaves distintas para cada ambiente. Sempre:
- Use uma **chave de teste** (`sk_test_...`) durante o desenvolvimento e o staging
- Use uma **chave live** (`sk_live_...`) apenas no seu ambiente de produção
- Nunca misture as duas — uma chave de teste não pode processar pagamentos reais, e uma chave live em um ambiente de teste é um risco desnecessário

### Rotacione as chaves regularmente

A rotação periódica de chaves limita a janela de dano caso uma chave seja silenciosamente comprometida. Rotacione sua chave ativa chamando:

```http
POST /api/api-keys/me/regenerate
X-Api-Key: sk_live_sua_chave_aqui
```

Após a rotação, atualize a chave em todos os ambientes e cofres de segredos imediatamente. A chave antiga é invalidada no momento em que a nova é emitida.

## Verificação de Webhooks

O CertoPay envia eventos de webhook (como `transaction.paid`) para seu endpoint quando o status do pagamento muda. No entanto, os payloads de webhook isoladamente não podem ser considerados prova de pagamento — um atacante poderia enviar um payload `PAID` forjado diretamente para seu endpoint via POST.

### Sempre verifique consultando a transação

Após receber qualquer webhook, confirme o status real da transação fazendo uma chamada autenticada à API:

```http
GET /api/transactions/{transactionId}
X-Api-Key: sk_live_sua_chave_aqui
```

Só conceda acesso ao produto ou serviço adquirido depois que a resposta da API confirmar que o `status` da transação é `PAID` ou `CAPTURED`.

> **Aviso:** **Nunca finalize um pedido baseando-se apenas no payload do webhook.** Sempre realize uma consulta no lado do servidor para confirmar o status da transação diretamente na API do CertoPay antes de entregar bens ou serviços.

### Requisitos do endpoint de webhook

- Seu endpoint de webhook **deve usar HTTPS**. O CertoPay não entregará eventos a endpoints em HTTP simples.
- Responda com HTTP `200` o mais rápido possível — faça o processamento pesado de forma assíncrona (colocar um job em fila, atualizar o BD, etc.) para evitar timeouts de entrega.
- Se seu endpoint retornar um status diferente de `2xx`, o CertoPay tentará reentregar o evento. Garanta que seu handler seja idempotente para que entregas duplicadas não causem entrega duplicada do pedido.

## Limites de Taxa

O CertoPay aplica um limite padrão de **100 requisições por minuto** por Chave de API para proteger a estabilidade do serviço. Exceder esse limite retorna uma resposta `HTTP 429 Too Many Requests`.

### Implemente backoff exponencial em respostas 429

Quando sua integração receber uma resposta `429`, não tente novamente imediatamente — isso só vai piorar a situação. Em vez disso, espere um tempo crescente exponencialmente entre as tentativas:

```javascript
async function withRetry(fn, retries = 3) {
  for (let i = 0; i < retries; i++) {
    try {
      return await fn();
    } catch (err) {
      if (err.status === 429 && i < retries - 1) {
        await new Promise(r => setTimeout(r, Math.pow(2, i) * 1000));
      } else throw err;
    }
  }
}
```

Isso espera 1 segundo antes da primeira tentativa, 2 segundos antes da segunda e 4 segundos antes da terceira, reduzindo a pressão sobre a API e dando à sua requisição a melhor chance de sucesso.

> **Dica:** Se sua integração regularmente se aproxima ou excede o limite de 100 req/min — por exemplo, em cenários de processamento de pagamentos em massa — contate o suporte do CertoPay para solicitar um limite de taxa maior para sua conta.

## Somente HTTPS

Toda comunicação com a API do CertoPay deve usar **HTTPS**. HTTP simples não é suportado e as conexões serão recusadas.

Isso se aplica a:
- Toda chamada de API de saída do seu servidor para `https://v2.certopaybrasil.com/api`
- Seu endpoint de webhook de entrada, que também deve ser servido via HTTPS

> **Nota:** Os certificados TLS no seu endpoint de webhook devem ser válidos e emitidos por uma Autoridade Certificadora confiável. Certificados autoassinados causarão falhas na entrega de webhooks.
